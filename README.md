# vLLM Inference Benchmarking Harness

Benchmarks LLM serving performance — throughput, latency percentiles, GPU
memory, and $/1M-token cost — for Llama-3.2-3B served with vLLM (fp16, AWQ,
GPTQ) versus a from-scratch naive HuggingFace `transformers.generate()`
baseline, on a single rented GPU. Includes an OpenAI-compatible serving
endpoint and a Prometheus/Grafana observability stack for live monitoring.
Solo, 3-week portfolio project.

**→ [Full results and writeup](reports/README.md)** — methodology,
throughput/latency/cost/memory charts, and headline findings, including a
kernel-configuration bug that was found and fixed mid-project (AWQ was
initially measured as the slowest config; corrected, it's the fastest).

## Quick summary

- vLLM is **1.7-3.0x faster per-token** than the naive baseline even with no
  batching advantage (concurrency=1), and **22-45x higher throughput** at
  realistic concurrency (32), once continuous batching compounds with that
  per-token speed advantage.
- The naive baseline uses **~3.1-3.3x less peak GPU memory**, since it never
  pre-reserves a KV-cache block pool — a real tradeoff, not just a deficiency.
- AWQ (with the correct `awq_marlin` kernel) is the fastest and cheapest
  config tested: **$0.034/1M output tokens** at concurrency=32, versus
  $0.057 (fp16), $0.069 (GPTQ), and $1.55 (naive baseline).

## Repo layout

- `serving/` — vLLM server launch script
- `baseline/` — naive HF `transformers.generate()` server (same OpenAI API contract)
- `benchmarks/` — load generator (vendored from vLLM), concurrency-sweep
  orchestration, GPU stats capture, cost model
- `observability/` — Prometheus/Grafana/GPU-exporter setup for live monitoring
- `results/` — raw benchmark output (git-ignored, regenerated by re-running
  the sweeps on rented hardware)
- `reports/` — the writeup, generated charts, and chart-generation script
- `dev/active/vllm-benchmarking/` — build plan, decisions log, task checklist

See `CLAUDE.md` for exact commands (environment setup, launching servers,
running sweeps, starting the observability stack) and environment/quirks
notes.

# vLLM Inference Benchmarking Harness

Benchmarks LLM serving performance — throughput, latency percentiles, GPU
memory, and $/1M-token cost — for Llama-3.2-3B served with vLLM (fp16, AWQ,
GPTQ) versus a from-scratch naive HuggingFace `transformers.generate()`
baseline, on a single rented GPU. Includes an OpenAI-compatible serving
endpoint and a Prometheus/Grafana observability stack for live monitoring.
Solo, 3-week portfolio project.

**→ [Full results and writeup](reports/README.md)** — methodology,
throughput/latency/cost/memory charts, and headline findings, including a
kernel-configuration bug that was found and fixed mid-project (AWQ was
initially measured as the slowest config; corrected, it's the fastest).

## Quick summary

- vLLM is **1.7-3.0x faster per-token** than the naive baseline even with no
  batching advantage (concurrency=1), and **22-45x higher throughput** at
  realistic concurrency (32), once continuous batching compounds with that
  per-token speed advantage.
- The naive baseline uses **~3.1-3.3x less peak GPU memory**, since it never
  pre-reserves a KV-cache block pool — a real tradeoff, not just a deficiency.
- AWQ (with the correct `awq_marlin` kernel) is the fastest and cheapest
  config tested: **$0.034/1M output tokens** at concurrency=32, versus
  $0.057 (fp16), $0.069 (GPTQ), and $1.55 (naive baseline).
- vLLM's continuous batching beats a static-like (`--max-num-seqs`-capped)
  config by **~15-19% throughput** at concurrency 16/32, with roughly
  **2x worse median TTFT** under the static-like config — see
  [Continuous vs. static batching](reports/README.md#continuous-vs-static-batching-awq-addendum).
- The findings above hold at a smaller model size too: re-run against
  Llama-3.2-1B-Instruct, AWQ is again the fastest/cheapest config
  (**$0.019/1M tokens** at concurrency=32) and GPTQ again degrades at high
  concurrency — see [Does this generalize?](reports/README.md#does-this-generalize-1b-vs-3b).
  One surprise: peak GPU memory barely differs between 1B and 3B, since
  vLLM's KV-cache reservation policy dominates it more than model size does.

## Repo layout

- `scripts/` — one-time environment setup and dataset download
- `serving/` — vLLM server launch script
- `baseline/` — naive HF `transformers.generate()` server (`launch_baseline.sh`
  execs `hf_inference_server.py`), same OpenAI API contract as vLLM
- `benchmarks/` — load generator (vendored from vLLM), concurrency-sweep
  orchestration (`run_matrix.sh`, `run_baseline_matrix.sh`,
  `run_batching_comparison.sh`), smoke test, GPU stats capture, cost model
- `observability/` — Prometheus/Grafana/GPU-exporter setup for live monitoring
- `results/` — raw benchmark output (git-ignored, regenerated by re-running
  the sweeps on rented hardware)
- `reports/` — the writeup (`README.md`), chart-generation script
  (`generate_charts.py`), and generated assets (`images/` — PNG charts +
  the live-dashboard GIF)

## Running it yourself

- One-time env setup on a fresh GPU box: `./scripts/setup_env.sh`
- Start vLLM: `./serving/launch_vllm.sh <model-repo> [awq_marlin|gptq] [port]`
- Run the full concurrency matrix: `./benchmarks/run_matrix.sh`
- Run the same matrix against a different model set (e.g. the 1B
  generalization check) without editing the script: set `CONFIGS_OVERRIDE`
  to newline-separated `name:model:quantization` entries, e.g.
  `CONFIGS_OVERRIDE=$'1b-fp16:meta-llama/Llama-3.2-1B-Instruct:\n1b-awq:AMead10/Llama-3.2-1B-Instruct-AWQ:awq_marlin\n1b-gptq:ModelCloud/Llama-3.2-1B-Instruct-gptqmodel-4bit-vortex-v2.5:gptq' ./benchmarks/run_matrix.sh`
- Run the continuous-vs-static batching comparison (AWQ only, concurrency
  8/16/32): `./benchmarks/run_batching_comparison.sh`
- Start the naive baseline: `./baseline/launch_baseline.sh <model-repo> [port]`
- Start the observability stack: `./observability/install_observability_stack.sh`
  then `launch_prometheus.sh` / `launch_gpu_exporter.sh` / `launch_grafana.sh`
- Regenerate the report charts (no GPU needed): `python3 reports/generate_charts.py`
